## 1. 全体構造とファイル間の依存関係

現状の各ファイルは、お互いに以下のような関係性で結びついています。

```
[ 開発・ビルドの司令塔 ]
  ├─ dev.js   (開発サーバー: ライブリロード、オンザフライでプレースホルダーを置換)
  └─ build.js (本番ビルド: 先行コンパイルした成果物をBase64化してメインに埋め込み)

[ ランタイム・コア（ブラウザ側） ]
  📌 main.ts ──────► SlideEngine2.ts (Markdown-itによるパース・描画)
    │
    └─► viewer.ts (メイン画面 UI 制御)
          │
          ├─ (window.open / srcdoc)
          │     ▼
          └─► presenter.ts (プレゼンター画面 制御 / メッセージ通信で相互同期)

```

---

## 2. 2つの実行モードにおける処理フローの全貌

現状の実装が「複雑」に見える最大の原因は、**`dev.js` と `build.js` で、ブラウザに渡すスクリプト（`main.js`）の中身を物理的に力技で書き換えている点**にあります。

### ① `bun run dev.js`（開発サーバー起動時）のフロー

開発時は、DX（開発者体験）を高めるために、ファイルをインライン化せず「分離された状態」で動かそうとしています。

```
[dev.js]
  │
  ├─► ./index.html の </body> 直前に「ライブリロード用WebSocketスクリプト」を動的挿入
  │
  └─► /dist/main.js へのリクエストをインターセプト
        │
        ├─ 開発用の生のHTMLテキスト（DEV_HTML: プレフィックス付き）を用意
        │  （このHTMLは内部で普通に <script src="/dist/presenter.js"> を参照している）
        │
        └─ main.js 内の「__PRESENTER_DATA_PLACEHOLDER__」をこの生HTMLで力づくで置換して配信

```

- **ブラウザ側の挙動**: `viewer.ts` がプレゼンター窓を開く際、埋め込まれたデータが `"DEV_HTML:"` で始まっているのを見て、**「あ、今は開発モードだな」と判断**し、Base64デコードをスキップしてそのまま `srcdoc` に流し込みます。

### ② `bun run build.js`（本番スタンドアロンビルド時）のフロー

本番ビルドでは、「ダブルクリックで動く単一HTML」を作るため、すべての依存アセットを1つのファイルにシュリンク（収縮）させます。

```
[build.js]
  1. presenter.ts をビルド ➔ dist/presenter.js 生成
  2. pptxExport.ts をビルド ➔ dist/pptxExport.js 生成
  3. main.css をビルド ➔ dist/main.css 生成
  4. src/presenter.html に「dist/presenter.js」と「dist/main.css」の中身を物理的にガバッとインライン注入
  5. 組み上がった「完全内包型 presenter.html」を、Bufferを使って【Base64文字列】に変換 🔒
  6. main.js 内の「__PRESENTER_DATA_PLACEHOLDER__」を、この【Base64文字列】で置換
  7. 最後に index.html 内の <link> や <script> タグを、完成した全てのCSS/JSの中身で全置換

```

- **ブラウザ側の挙動**: `viewer.ts` はデータが `"DEV_HTML:"` で始まっていないため、**「本番ビルド版だな」と判断**。`atob()` でBase64をデコードし、`Blob` を作って `URL.createObjectURL(blob)` を経由してポップアップウィンドウに流し込みます。

## 3. 現状のコードが抱える「3つの密結合（環境依存）」

初見でコードを読んだ際につまずきやすい、**密結合になっているポイント**です。これらが今回リファクタリングで一掃される対象となります。

| 密結合のポイント         | 現状の実装方法                                                                                                                               | 発生している問題                                                                                                          |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **① スタイルの読み込み** | `window.location.hostname` が `localhost` かどうかで `isDev` フラグを立てて、`<link>` か `<style>` かをランタイムで分岐。                    | 実行環境（ドメイン名など）にコードが依存してしまっている。                                                                |
| **② アセットの展開**     | 文字列の先頭が `"DEV_HTML:"` かどうかで、生のまま扱うか `atob()` でBase64デコードするかを分岐。                                              | `viewer.ts` の中に、ビルドシステムの都合（Base64）や開発サーバーの都合（DEV_HTML）のロジックが漏れ出している。            |
| **③ ビルトインテーマ**   | 開発時もビルド時も、すべてのテーマCSSの中身を巨大なJSONオブジェクト（`builtinThemes`）にして、`main.js` のプレースホルダーへ丸ごと埋め込み。 | 開発サーバー経由（HTTP）で動かす時であっても、使わないテーマを含めたすべてのCSSがJSに強制同梱され、無駄に重くなっている。 |
