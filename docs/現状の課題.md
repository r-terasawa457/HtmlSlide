これまでのリファクタリングによって、状態管理の分離（`AppState` / `ViewerState` / `PresenterState`）と印刷周りの宣言的制御は極めて強固なものになりました。

この洗練された土台の上で、アーキテクチャを完全に完成させるために「次に着手すべき残る課題」を4つのポイントに整理しました。

---

## 1. メッセージングの型安全化（通信定義の脆弱さ）

現在、親ウィンドウとプレゼンターウィンドウの間での通信（`postMessage`）は、生のオブジェクトと文字列トークン（`"sync_page"` や `"presenter_ready"`）で行われています 。

- **問題点**: イベント名にタイポがあってもコンパイルエラーにならず、実行時までバグを検知できません。また、受け取り側（`handleMessage`）の `e.data` が `any` 状態であるため、型補完が効きません 。

- **対策案**: メッセージのインターフェースを判別共用体（Discriminated Unions）として定義し、双方のウィンドウで共通の型定義をインポートして利用する設計へ進化させます。

```typescript
// 例: src/states/MessageProtocol.ts などの新設
export type SlideMessage =
  | { type: "presenter_ready" }
  | { type: "presenter_init"; slidesHtml: string; page: number }
  | { type: "sync_page"; page: number };
```

---

## 2. 初期化・接続シーケンスのコンポーネントからの剥離

ページの日常的なめくり同期は `App.svelte` の `$effect` に中央集約されましたが、**最初の立ち上げ時のハンドシェイク（接続確立）ロジック**がまだ `ViewerCore.svelte` の中に命令的に残っています 。

- **問題点**: `ViewerCore` が「プレゼンター窓が開いたこと（`presenter_ready`）」を検知し、`AppState` にあるHTMLデータを引っ張り出して送り返すという、通信のハブのような役割を担ってしまっています 。これではUIコンポーネントの責務を超過しています。

- **対策案**: 親ウィンドウ側のメッセージ受信窓口（`window.addEventListener("message")`）もコンポーネントから完全に剥がし、`AppState` の初期化時（または専用の通信管理クラス）へ移譲します。UIコンポーネントは純粋に描画とローカルのイベント検知だけに専念させるべきです。

---

## 3. 【重要】別ウィンドウにおけるアセット・画像の参照保証

`SlideIframe` を共通化したことで、内部で `AssetProvider.resolveStyleTag("src/css/slide_root.css")` を呼び出す構造に変わりました。ここで一つ、マルチウィンドウ特有の潜在的なリスクが生じています。

- **問題点**: プレゼンター画面（別窓）は完全に独立した別プロセス・別メモリ空間です 。開発環境（HTTP駆動）では `fetch` でCSSを取得できるため動きますが、製品環境（単一HTML動作）において `AssetProvider` が親ウィンドウのメモリ（`globalThis.EmbeddedAssets` など）に依存している場合、**プレゼンター側で `SlideIframe` 内のCSS読み込みや、ドロップされたローカル画像（`assetsMap`）の解決に失敗するリスク**があります 。

- **対策案**: `slidesHtml` だけでなく、ドロップされたアセットマップ（`assetsMap`）や解決済みのCSS自体も、最初の `presenter_init` メッセージのペイロードに含めて親から子（プレゼンター）へ明示的に手渡す（パッシングする）設計、あるいは `window.opener` 経由で親の静的データに安全にアクセスするプロキシを `AssetProvider` 側に仕込む必要があります 。

---

## 4. 将来的なコンテンツ（HTML）の動的リアルタイム同期

現在の設計は、「起動時に一度ドロップしたら、閲覧中にMarkdownの内容は変わらない」という前提に基づいているため、`slidesHtml` のプレゼンターへの同期は起動時の1回（`presenter_init`）しか走りません 。

- **問題点**: 今後、システムを発展させて「メイン画面側でMarkdownをリアルタイム編集しながら、プロジェクター（プレゼンター窓）側の表示を即座に書き換える（ライブプレビュー機能）」といった拡張を行う際、現在のメッセージング構造では対応できません。
- **対策案**: ページ同期と同様に、親の `AppState.slidesHtml` の変更を検知する `$effect` を定義し、変更があったら自動的に `{ type: "update_content", html: ... }` をプレゼンター側にブロードキャストするリアクティブな通信パイプラインをあらかじめ見据えておく必要があります。
